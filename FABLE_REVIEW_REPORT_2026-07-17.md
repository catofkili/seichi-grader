# Seichi Grader — Fable 审阅报告

日期：2026-07-17  
站点：<https://catofkili.github.io/seichi-grader/>  
当前发布提交：`1293b3c`  
项目属性：纯前端、静态 GitHub Pages；图片和模型推理均在访客浏览器本地完成。

## 1. 产品目标

这是一个面向动画圣地巡礼的工具。用户上传动画截图与实景照片后，可以：

1. 将动画截图的影调、色彩迁移到实景照片；
2. 通过构图对齐和现场取景，让实景与动画画面对应；
3. 抠出动画人物并合成到实景中；
4. 导出调色图、对比图、对比动图/GIF、角色透明 PNG 与 LUT；
5. 在多张实景照片中寻找与动画截图最相近的一张。

站点署名为 **LovelySummer Channel**，反馈邮箱为 `lovelysummerchannel@outlook.com`。

## 2. 当前技术结构

- `index.html`：页面结构与流程入口。
- `app.js`：上传、调色预览、角色合成、导出、圈选交互等主逻辑。
- `color.js`：CDF 影调匹配、MKL/Lab 色彩迁移、天空/地景分区、Bloom、LUT。
- `segment.js`：轻量、无需模型下载的算法抠图。
- `ai-segment.js`、`detect.js`、`sam-segment.js`：人物检测 → 局部抠图 → SAM 兜底的模型路径。
- `ai-worker.js`：模型推理 Worker，避免占用页面主线程。
- `ort-env.js`：ONNX Runtime 与模型分块下载、会话管理。
- `camera/`：现场取景拍摄与动画参考图叠加。
- `sw.js`：程序与模型缓存；当前应用缓存版本为 `seichi-app-v10`。

## 3. 已实现功能与用户提出的改动

### 3.1 页面发现性、文案与移动端

- 加入 SEO `meta description`：说明可从动画截图提取色调与配色，并套用到实景圣地巡礼照片。
- “角色合成”区默认展开并强化展示，后续按用户要求移除了尴尬的“重点功能”四字标签。
- 修复角色合成区右侧信息按钮在手机端挤压、变形的问题。
- 底部增加开发者信息、GitHub、隐私说明和反馈邮箱。
- Android 上传图片时改用 `image/*`，优先唤起图库/照片选择器，而非文件管理器。

### 3.2 现场取景与构图对齐

- 现场取景支持参考动画图旋转 90°；解决横向动画截图进入竖屏拍摄时方向不对的问题。
- 参考图叠加按实际视频可见区域对齐，拍摄结果会旋转回正确方向。
- 提供洋葱皮构图对齐：照片与动画图可拖拽、缩放、裁剪至同一构图。

### 3.3 主预览的三种对比方式

已提供三种可切换模式：

1. **滑动**：原图与调色后实景的滑动对比。
2. **叠加**：动画截图覆盖调色后的实景，用户可调动画透明度。
3. **上下**：动画截图在上、调色实景在下，保留各自比例。

叠加和上下预览使用“动画截图 vs 调色后实景”，而不是“原图 vs 调色后实景”。

### 3.4 模型下载、网络与诊断

- 模型文件按需下载，基础调色不需要 AI 模型。
- 已内置“检测模型下载线路”：检测本站人物检测模型、ISNet 分块、jsDelivr ONNX Runtime 的 HTTP 状态、耗时和大小，并可导出诊断 JSON。
- 已向用户说明模型下载可能受网络、地区网络策略、缓存、不同浏览器/设备影响。
- Service Worker 对模型采用缓存；版本化页面链接（`?v=<commit>`）只强制刷新页面资源，不会让同一浏览器重新下载模型。

### 3.5 缓存与 GitHub Pages 发布

- 站点通过 GitHub Pages 从 `main` 自动发布。
- 因用户多次遇到“刷新后仍没看到更新”，每次重要发布均提供常规链接与带提交短哈希的强制刷新链接。
- 程序页面、脚本、样式和 Service Worker 缓存版本已多次同步更新，避免“新 HTML + 旧脚本”混搭。

### 3.6 模型抠像性能与稳定性

发现的问题：电脑端“模型抠像”原先直接在页面主线程运行。ONNX 推理、1024px 预/后处理和遮罩处理会阻塞页面；用户在卡顿时再点其它控件，标签页会显示未响应。

已实施的修复：

- 自动“模型抠像”在所有平台都改为一次性 Worker 执行；不再只给 iPhone/iPad 使用 Worker。
- “圈选抠像”模型路径也迁移至 Worker。
- Worker 完成后即释放 ONNX 会话并终止，降低长时间使用后的内存压力。
- 主线程只保留结果合并、遮罩微调和预览绘制。

## 4. 抠像交互：当前状态

### 4.1 三条抠像路线

1. **模型抠像**：自动检测人物，效果优先；多人/小人使用检测 → 局部 ISNet → SAM 兜底。
2. **圈选抠像**：用户指定人物范围，仍走模型路径，帮助模型准确定位。
3. **算法抠像**：不下载模型。现在不再把整张动画当作主体猜测，而是先进入圈选，只对用户指定区域运行轻量算法。

### 4.2 最新两步圈选工作流

用户指出原引导不清晰，也指出“蓝色强制保留”不应在算法识别前预画。当前交互已改为：

1. **第一步 · 圈出识别范围**  
   用粉色矩形、圆形或自由画笔圈区域；文案明确要求“请将目标完全包含在框内”。

2. **执行识别**  
   算法/模型只对第一步的范围处理。识别完成后，弹窗不关闭。

3. **第二步 · 强制保留（可选）**  
   弹窗展示完整动画截图；已识别目标保留原样，未识别部分覆盖深色半透明蒙版。此时粉色选区工具收起，出现蓝色“强制保留”画笔与粗细滑杆。用户刷出的部分会立刻并入目标的 alpha 遮罩。

4. **完成抠像**  
   用户确认后关闭窗口，可继续调阈值、收边、拖拽、合成或导出。

实现细节：蓝色强制保留数据保存为 `state.forcedAlpha`，在后续阈值和边缘收紧后仍覆盖回最终遮罩；橡皮擦会同时清除普通遮罩和强制保留遮罩。

## 5. GIF 与动图导出

已有导出：

- 滑动对比视频（WebM/MP4，依浏览器支持而定）。
- 动画截图渐变到实景的 GIF，约两秒，结束停在实景，不循环。

曾出现的 GIF 问题及修复：

1. 首版 GIF 文件头与帧结构存在，但 LZW 位宽从 9 位到 10 位时提前切换，解码在每帧早期中断，导致 GIF 损坏。
2. 已将位宽升级改为正确的延后一项，使用随机像素编码→解码校验。
3. 用户认为修复后 GIF 仍偏模糊。已将最长边从 720px 提高至 1080px，并将粗糙的 3-3-2 固定色表替换为 6×6×6 色立方加 40 阶灰度，优先改善天空、阴影、线稿和边缘层次。

约束：GIF 格式每帧最多 256 色，因此不可能完全达到 PNG/JPEG 的连续色彩；更高质量可考虑保留 MP4/WebM 作为“高清动图”选项。

## 6. 当前最新的、尚未实现的抠像改进需求（本次请重点审阅）

用户的最新观察：许多动画角色会出现“外轮廓被选中，但轮廓内部部分没有被选中”的情况，尤其是人物的脸、手、服装内部。例如：人物外轮廓近似闭合，但内部被算法视为背景/透明，导致角色看起来被挖空。

用户提出两项方向，其中第一项是算法层优化，第二项是网页交互层补救：

### A. 算法：对接近闭合轮廓的内部更宽松

目标：当算法已形成完整或接近完整的角色边界时，内部区域不要因为局部颜色、反光、线稿或背景相似而被挖掉。

建议 Fable 审阅：

- 是否应在圈选局部的 alpha mask 上做“保守填洞”（hole filling）？
- 对非完全封闭轮廓，可否使用形态学闭运算、距离阈值、连通域约束或颜色/边缘连续性来有限度地补全？
- 如何避免把角色缝隙、镂空服装、背景窗口等真实透明区域错误填满？
- 是否应将这一策略限定在“算法抠像 + 用户圈选区域”，而不是应用于模型抠像的所有结果？

### B. 网页交互：点击蒙版阴影块，将整个块纳入目标

目标：在“完整图片 + 非目标深色蒙版”的第二步里，用户不必总是手工刷脸/手的一整片。用户点击一块连通的暗色区域后，该区域应作为一个整体被识别/并入当前目标。

建议交互：

1. 在第二步加一个“点选阴影块”工具（可与蓝色笔刷并列）。
2. 用户点击蒙版暗区时，计算该点所在的连通域；高亮预览后，一次点击确认纳入目标。
3. 该功能应优先在第一步粉色范围内部生效，避免用户误点整张背景。
4. 可设置最大面积、安全阈值和撤销，防止把大块天空/地面误并入角色。

这两项尚未实施；当前蓝色强制保留笔刷是可靠的手工补救方案。
